# GitHub Issue 作成ウィザード

AskUserQuestion を使い、疑問点が解消するまでヒアリングしてから GitHub Issue を作成する。
作成完了後は次の Issue を作るか確認し、する場合は最初から繰り返す。

## ループ手順

### Step 1: 作業内容をヒアリング

AskUserQuestion で聞く:
「どんな作業をしたいですか？（ざっくりした説明でOK）」

回答からIssueタイプを判定する:
- 不具合・おかしな挙動 → **バグ修正**（label: bug, title prefix: `fix: `）
- 新しい機能・画面 → **機能追加**（label: enhancement, title prefix: `feat: `）
- コード整理・改善 → **リファクタリング**（label: refactoring, title prefix: `refactor: `）

判定が曖昧な場合は AskUserQuestion で確認する。

---

### Step 2: タイプ別ヒアリング

情報が不足している項目を AskUserQuestion で順番に確認する。
**1回の質問で複数項目をまとめて聞いてよい。**
任意項目は「分からなければスキップOK」と伝える。

#### バグ修正の場合

必須:
- 対象画面（下記から選択、または自由記述）
- 再現手順（ステップ形式）
- 期待する動作
- 実際の動作

任意:
- 対象画面URL（ブラウザのアドレスバーから）
- エラーログ・スタックトレース
- 制約事項（APIの仕様は変えない / UIの見た目は変えない / 特定ファイルは触らない）

#### 機能追加の場合

必須:
- 対象画面・モジュール（下記から選択、または自由記述）
- 背景・目的
- 実装したい機能
- 完了条件（箇条書き）

任意:
- 対象画面URL
- 参考にすべき既存実装（例: 受注フォームと同じ構造で）
- 技術的なスコープ（DBスキーマ変更あり / APIエンドポイント追加・変更あり / 新規コンポーネント / 既存コンポーネント変更）
- その他の注意点

#### リファクタリングの場合

必須:
- 対象モジュール（下記から選択、または自由記述）
- 対象画面URL（複数あれば列挙）
- リファクタリングの目的（可読性向上 / 重複排除 / パフォーマンス改善 / 型安全性向上 / テスタビリティ向上）
- 現状の問題点と目標の状態

任意:
- 変えてはいけないこと（外部APIの仕様 / UIの見た目・操作感 / 既存テストの期待値 / DBスキーマ）
- その他の制約・補足

---

#### 画面・モジュールの選択肢（参考）

売上（/sales/sales）/ 受注（/sales/orders）/ 見積（/sales/quotations）/
仕入（/purchase/purchases）/ 発注（/purchase/orders）/
請求（/accounting/billing）/ 入金・支払（/accounting/payments）/ 締め処理（/accounting/closing）/
在庫（/warehouse/inventory）/ 棚卸（/warehouse/stocktake）/ 出荷（/warehouse/shipments）/
マスタ設定（取引先）/ マスタ設定（商品）/ マスタ設定（その他）/
ダッシュボード / 共通・レイアウト / API・バックエンド / その他

---

### Step 3: Issue 本文を組み立てる

収集した情報を Markdown で整形する。
タイプ別のフォーマット:

**バグ修正:**
```
## 対象画面
{画面名}（{URL}）

## 再現手順
1. ...
2. ...

## 期待する動作
...

## 実際の動作
...

## エラーログ
（あれば）

## 制約事項
（あれば）
```

**機能追加:**
```
## 対象画面 / モジュール
{画面名}（{URL}）

## 背景・目的
...

## 実装したい機能
...

## 完了条件
- [ ] ...
- [ ] ...

## 技術的なスコープ
- [ ] DBスキーマ変更あり（マイグレーション必要）
- [ ] APIエンドポイント追加・変更あり
- [ ] 新規コンポーネント作成
- [ ] 既存コンポーネントの変更

## 参考にすべき既存実装
（あれば）

## その他の技術的な注意点
（あれば）
```

**リファクタリング:**
```
## 対象モジュール / URL
{モジュール名}（{URL}）

## 目的
{可読性向上 など}

## 現状 → 目標
...

## 変えてはいけないこと
- [ ] 外部APIの仕様（リクエスト/レスポンス形式）
- [ ] UIの見た目・操作感
- [ ] 既存テストの期待値
- [ ] DBスキーマ

## その他の制約・補足
（あれば）
```

---

### Step 4: Issue を作成する

本文の末尾に必ず以下の署名を追加する:

```
---
🤖 *この Issue は Claude によって自動作成されました*
```

```bash
gh issue create \
  --title "{prefix}{タイトル}" \
  --body "{組み立てた本文}

---
🤖 *この Issue は Claude によって自動作成されました*" \
  --label "{label}"
```

作成後、Issue の URL をユーザーに伝える。

---

### Step 5: 継続確認

AskUserQuestion で聞く:
「他にも Issue を作りますか？」

- Yes → Step 1 に戻る
- No → 終了
